# 路线图：Detent

## 概述

Detent 是一个基于控制论的多智能体编排 harness 框架，运行于 Claude Code 内。路线图遵循架构依赖链中的 7 层构建顺序：状态基础设施必须先于技能调用它，技能必须先于智能体模板在其中运行，智能体模板必须产出稳定产物后评估器才能评判，完整流水线必须可靠运行后监控仪表盘才有价值。三个里程碑对应三个交付边界：M1 引擎（可用的流水线骨架）、M2 智能体（完整约束传播和质量机制）、M3 Web UI（可视化层）。

## 里程碑

- 📋 **M1 引擎** — 阶段 1-2（状态基础设施 + 流水线骨架）
- 📋 **M2 智能体** — 阶段 3-5（真值面、对抗式智能体、编码循环、可观测性）
- 📋 **M3 Web UI** — 阶段 6（可视化与配置层）

## 阶段列表

- [ ] **阶段 1：状态基础设施** — CLI 工具、.detent/ 数据结构、setup 技能 — 一切的基础
- [ ] **阶段 2：流水线骨架** — 5 个工作流技能和双模式门控架构 — 在智能体存在之前验证端到端会话桥接
- [ ] **阶段 3：真值面与规划智能体** — 约束账本、冻结决策、对抗式规划智能体模板（D/G/H/J）
- [ ] **阶段 4：编码循环与恢复** — 编码者/评估者对抗循环、跨阶段重入、告警信号路由
- [ ] **阶段 5：可观测性** — Gemini CLI 异步行为日志、原始 JSONL 采集、语言偏好输出
- [ ] **阶段 6：Web UI** — 流水线配置页面和实时运行仪表盘，支持多流水线并行监控

## 阶段详情

### 阶段 1：状态基础设施
**目标**：detent-tools.cjs CLI 和 .detent/ 目录结构存在、正确、可被所有下游技能调用
**依赖**：无（第一阶段）
**需求覆盖**：ENG-01, ENG-02, ENG-03, ENG-04
**成功标准**（必须为真）：
  1. 在目标仓库中运行 `node detent-tools.cjs` 不带参数，打印使用说明并正常退出
  2. `/detent:setup` 引导用户完成配置问答，在目标仓库创建 .detent/config.json、.detent/state.json 和 .detent/truth-surface/
  3. 调用 detent-tools.cjs 写入有效 JSON 后，原子更新 .detent/state.json，执行 /clear 后文件内容持久正确
  4. .detent/state.json 的 schema 从第一天起就包含 pipeline stage、current unit、iteration count 和 reentry_depth 字段
  5. .detent/config.json 存储 mode、model budget、locale、流水线阶段开关和 unit 粒度 — 可在任何智能体会话外读取和编辑

### 阶段 2：流水线骨架
**目标**：5 个工作流技能存在，端到端流水线可运行 — 从 discovery 到 achieve — 具备双模式门控行为和会话连续性，在生产智能体模板加入之前
**依赖**：阶段 1
**需求覆盖**：PIPE-01, PIPE-02, PIPE-03, PIPE-04
**成功标准**（必须为真）：
  1. 按序调用 `/detent:discovery` 到 `/detent:achieve`，每个技能在启动时读取 .detent/state.json，完成时更新它
  2. 在 supervised 模式下，流水线在定义的门控点暂停，用户确认后正确恢复
  3. 在 autonomous 模式下，同一流水线运行时不在任何门控点暂停
  4. 流水线中途 /clear 后，调用下一个技能从正确的状态位置恢复（而非从头开始）
  5. 子进程生成器应用 4 层隔离（作用域工作目录、.git/HEAD 屏蔽、空插件目录、仅项目设置）— 通过运行流水线并确认无 50K token 重注入开销来验证

### 阶段 3：真值面与规划智能体
**目标**：约束传播是真实的 — 冻结决策存在于 .detent/truth-surface/ 中，所有规划阶段的智能体模板已定义，对抗式规划流水线（D → G-红/蓝 → H → J）可端到端运行
**依赖**：阶段 2
**需求覆盖**：TRUTH-01, TRUTH-02, TRUTH-03, PLAN-01, PLAN-02, PLAN-03, PLAN-04, PLAN-05
**成功标准**（必须为真）：
  1. .detent/truth-surface/ 包含 constraint-ledger.md、frozen-decisions.md 和 domain-model.md，具有 FROZEN/PROPOSED/SUPERSEDED 区域；标记为 FROZEN 的决策不能被任何智能体覆写
  2. D-批评智能体模板被调用后，针对需求集产出结构化批评，至少攻击一项需求与真值面的对齐性
  3. G-红智能体模板对 D-批评的输出产出真正的攻击（而非赞同）；G-蓝智能体模板产出结构化防御；两者输出结构化且可操作
  4. H-审查智能体模板判定编码就绪性，发出批准或拒绝，拒绝时附带明确的 reentry_stage 字段
  5. J-编译智能体模板产出可执行的代码交付文档，编码者智能体可直接据此行动，无需额外澄清

### 阶段 4：编码循环与恢复
**目标**：编码者/评估者对抗循环以机器结构化反馈运行，重入机制将矛盾作为新的冻结约束传播，告警信号在触发时升级至人工
**依赖**：阶段 3
**需求覆盖**：CODE-01, CODE-02, CODE-03, CODE-04, RECOV-01, RECOV-02, RECOV-03, RECOV-04
**成功标准**（必须为真）：
  1. 编码者智能体逐个执行实现单元；当评估者返回 FAIL 并附带 `[file:line] expected X, got Y` 反馈时，编码者携带该精确反馈重试 — 最多 5 次迭代
  2. PASS 时，单元被 git 提交，.detent/state.json 自动推进到下一个单元
  3. 任何智能体触发告警信号时，流水线立即停止并将矛盾呈现给人工 — 绕过所有正常门控逻辑
  4. 评估者触发的跨阶段重入回滚到指定的规划子阶段，并将矛盾作为新的冻结约束注入 constraint-ledger.md
  5. reentry_depth 达到 2 时，进一步的重入触发人工升级，而非另一次回滚

### 阶段 5：可观测性
**目标**：每次流水线运行异步产出结构化行为日志，不侵入核心智能体提示词，所有输出尊重语言偏好设置
**依赖**：阶段 4
**需求覆盖**：OBS-01, OBS-02, OBS-03
**成功标准**（必须为真）：
  1. 任何流水线阶段完成后，.detent/raw/ 包含该阶段的 JSONL 文件，记录 stream-json 输出和工具调用 — 智能体提示模板内无任何日志代码
  2. 对原始 JSONL 文件运行 Gemini CLI hook 后，在 .detent/logs/ 产出结构化行为日志 — 确认异步侧信道端到端可用
  3. config.json 中 locale 设为 zh-CN 时，所有面向用户的输出（日志、真值面文档、阶段产出物）以中文呈现

### 阶段 6：Web UI
**目标**：本地 Web 仪表盘允许用户可视化配置流水线并实时监控一个或多个运行中的流水线 — 双通道状态调和确保 stream-json 和 file-watch 不会破坏持久化状态
**依赖**：阶段 5
**需求覆盖**：UI-01, UI-02, UI-03, UI-04
**成功标准**（必须为真）：
  1. 打开仪表盘显示流水线配置页面，用户可以切换阶段开关、编辑配置值、关联仓库 — 变更通过 detent-tools.cjs 持久化到 .detent/config.json
  2. 运行中的流水线在浏览器中显示实时进度（当前阶段、当前单元、迭代次数），无需刷新页面
  3. 两个并行运行的流水线各自显示为仪表盘中独立的行，状态隔离 — 任一流水线的状态不会渗入另一个
  4. 如果 stream-json 通道中途停止，仪表盘回退到对 .detent/ 的文件监听，继续显示正确状态
